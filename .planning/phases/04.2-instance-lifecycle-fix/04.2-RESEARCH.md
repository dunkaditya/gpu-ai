# Phase 4.2: Instance Lifecycle State Machine Fix - Research

**Researched:** 2026-02-24
**Domain:** Go concurrency / provider status polling / HTTP middleware security
**Confidence:** HIGH

## Summary

Phase 4.2 closes the "Instance Becoming Running" broken flow identified in the v1.0 Milestone Audit. The root cause is two independent breaks: (1) `progressStatus` goroutine only transitions `creating -> provisioning` and never polls the provider to detect when the upstream pod is running, so instances are stuck in `provisioning` forever; (2) the `/internal/instances/{id}/ready` callback endpoint is protected by `LocalhostOnly` middleware, making it unreachable from cloud-init running on a remote GPU instance.

The fix requires three coordinated changes: extend `progressStatus` to poll `provider.GetStatus` in a loop until the upstream reports "running", then transition the instance to `booting`; replace `LocalhostOnly` middleware on the ready callback with per-instance `internal_token` authentication so GPU instances can call back over the network; and ensure the full lifecycle chain works end-to-end: creating -> provisioning -> booting -> (cloud-init callback) -> running.

**Primary recommendation:** Implement a polling loop in `progressStatus` that calls `provider.GetStatus` every 5 seconds (with exponential backoff on errors), transitions to `booting` when the provider reports "running", and replace `LocalhostOnly` with `InternalTokenAuth` on the ready/health callback routes. The engine needs access to the provider registry so `progressStatus` can look up the correct provider adapter by `upstream_provider` name.

### Pattern 1: Polling Loop in progressStatus

**What:** Extend `progressStatus` from a single-shot `creating->provisioning` transition to a polling loop that also drives `provisioning->booting` by querying the upstream provider.

**When to use:** When the upstream provider does not support webhooks and the only way to detect status changes is polling.

**Current code (broken):**
```go
func (e *Engine) progressStatus(instanceID string) {
    ctx := context.Background()
    // Only does creating -> provisioning, then exits
    updated, err := e.db.UpdateInstanceStatus(ctx, instanceID, StateCreating, StateProvisioning)
    // ... done. Instance stuck in provisioning forever.
}
```

**Fixed pattern:**
```go
func (e *Engine) progressStatus(instanceID string) {
    ctx := context.Background()

    // Step 1: Transition creating -> provisioning (immediate, as before)
    updated, err := e.db.UpdateInstanceStatus(ctx, instanceID, StateCreating, StateProvisioning)
    if err != nil {
        e.logger.Error("failed to progress to provisioning", ...)
        return
    }
    if !updated {
        e.logger.Warn("status already changed from creating", ...)
        return
    }

    // Step 2: Look up instance to get upstream_provider and upstream_id
    inst, err := e.db.GetInstance(ctx, instanceID)
    if err != nil {
        e.logger.Error("failed to get instance for status polling", ...)
        return
    }

    // Step 3: Look up provider adapter
    prov, ok := e.registry.Get(inst.UpstreamProvider)
    if !ok {
        e.logger.Error("provider not found for status polling", ...)
        return
    }

    // Step 4: Poll provider until running or terminal/timeout
    ticker := time.NewTicker(5 * time.Second)
    defer ticker.Stop()
    timeout := time.After(10 * time.Minute) // safety timeout

    for {
        select {
        case <-timeout:
            e.db.SetInstanceError(ctx, instanceID, "provisioning timeout: instance did not start within 10 minutes")
            return
        case <-ticker.C:
            status, err := prov.GetStatus(ctx, inst.UpstreamID)
            if err != nil {
                e.logger.Warn("status poll failed", ...)
                continue // retry on next tick
            }

            if status.Status == "running" {
                // Provider reports pod is running -> transition to booting
                // (cloud-init is now executing on the instance)
                updated, err := e.db.UpdateInstanceStatus(ctx, instanceID, StateProvisioning, StateBooting)
                if err != nil {
                    e.logger.Error("failed to transition to booting", ...)
                }
                if updated {
                    // Publish SSE event
                    // ... broker.Publish(instanceID, event)
                }
                return // polling complete; ready callback handles booting->running
            }

            if status.Status == "terminated" || status.Status == "error" {
                e.db.SetInstanceError(ctx, instanceID, "upstream instance failed to start")
                return
            }
        }
    }
}
```

**Key design decisions:**
- 5-second poll interval balances responsiveness vs API rate limiting (RunPod allows ~100 req/s)
- 10-minute timeout prevents leaked goroutines for stuck provisions
- Continue on poll error (transient network issues) rather than failing
- Publish SSE event on booting transition so customers see status change in real time
- Return after booting transition -- the ready callback handles the rest

### Pattern 2: Token-Based Auth for Internal Callbacks

**What:** Replace `LocalhostOnly` middleware on `/internal/instances/{id}/ready` and `/internal/instances/{id}/health` with per-instance `internal_token` authentication. The cloud-init script already sends `Authorization: Bearer <internal_token>` in the callback request.

**Current code (broken):**
```go
// server.go -- callback is LocalhostOnly, unreachable from GPU instances
s.mux.Handle("POST /internal/instances/{id}/ready",
    LocalhostOnly(InternalAuthMiddleware(deps.Config.InternalAPIToken,
        http.HandlerFunc(s.handleInstanceReady))))
```

**Fixed pattern:**
```go
// server.go -- callback uses instance token auth, reachable from GPU instances
s.mux.Handle("POST /internal/instances/{id}/ready",
    InstanceTokenAuth(deps.DB, http.HandlerFunc(s.handleInstanceReady)))
s.mux.Handle("POST /internal/instances/{id}/health",
    InstanceTokenAuth(deps.DB, http.HandlerFunc(s.handleInstanceHealth)))
```

**New middleware:**
```go
// InstanceTokenAuth validates the Authorization header against the instance's
// stored internal_token. This replaces LocalhostOnly for callback endpoints
// that must be reachable from GPU instances over the network.
func InstanceTokenAuth(dbPool *db.Pool, next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        instanceID := r.PathValue("id")
        if instanceID == "" {
            writeProblem(w, http.StatusBadRequest, "missing-id", "Instance ID is required")
            return
        }

        // Extract bearer token from Authorization header
        auth := r.Header.Get("Authorization")
        if !strings.HasPrefix(auth, "Bearer ") {
            writeProblem(w, http.StatusUnauthorized, "unauthorized", "Authorization required")
            return
        }
        token := strings.TrimPrefix(auth, "Bearer ")

        // Look up instance and verify token
        inst, err := dbPool.GetInstance(r.Context(), instanceID)
        if err != nil {
            if errors.Is(err, db.ErrNotFound) {
                http.NotFound(w, r)
                return
            }
            writeProblem(w, http.StatusInternalServerError, "internal-error", "Failed to verify token")
            return
        }

        if inst.InternalToken == nil || token != *inst.InternalToken {
            writeProblem(w, http.StatusForbidden, "forbidden", "Invalid instance token")
            return
        }

        next.ServeHTTP(w, r)
    })
}
```

**Security considerations:**
- Each instance gets a unique 16-byte hex token (32 characters) generated at provision time
- Token is included in the cloud-init bootstrap script and sent via HTTPS
- Token is stored in the `internal_token` column of the instances table
- This is MORE secure than `LocalhostOnly` + global `InternalAPIToken` because each instance can only authenticate itself, not other instances
- The handler's existing instance-specific token check (query param `?token=`) becomes redundant since the middleware already verified the token -- but keeping it is harmless

### Pattern 3: SSE Event Publication on Booting Transition

**What:** When `progressStatus` transitions an instance from `provisioning` to `booting`, publish an SSE event so connected clients see the state change in real time.

**Challenge:** `progressStatus` runs on `Engine` which doesn't have access to the `StatusBroker` (lives on `Server`). Two options:

**Option A (recommended): Pass broker to engine via EngineDeps**
```go
type EngineDeps struct {
    // ... existing fields ...
    StatusBroker StatusPublisher  // interface to decouple from api package
}
```

**Option B: Use a callback function**
```go
type EngineDeps struct {
    // ... existing fields ...
    OnStatusChange func(instanceID, status string)
}
```

Option A is cleaner because it uses the established dependency injection pattern. Define a `StatusPublisher` interface in the provision package to avoid circular imports:
```go
// StatusPublisher publishes instance status events.
// Implemented by api.StatusBroker.
type StatusPublisher interface {
    Publish(instanceID string, event any)
}
```

However, this introduces a coupling concern since StatusEvent is defined in the api package. Option B avoids this entirely with a simple callback. Given the project convention of minimal deps, Option B is simpler and recommended.

### Anti-Patterns to Avoid
- **Polling without timeout:** Every polling goroutine MUST have a maximum lifetime to prevent goroutine leaks. Use `time.After(10*time.Minute)` as safety net.
- **Polling without error handling:** Don't abort polling on transient errors. Log and continue to next tick.
- **Blocking on DB in middleware:** The `InstanceTokenAuth` middleware does a DB lookup per request. This is acceptable for callback endpoints (low frequency, ~1 call per instance lifetime) but would be wrong for high-throughput routes.
- **Modifying cloud-init:** The bootstrap script already sends `Authorization: Bearer <internal_token>`. Do NOT change the callback URL or authentication mechanism in the template -- only change the server-side middleware.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Exponential backoff | Custom backoff logic | Simple counter with `time.Duration(1<<attempt)*time.Second` | Already used in RunPod client retry logic; keep consistent |
| Provider status polling | Custom scheduler | `time.Ticker` in goroutine | Stdlib is sufficient for single-instance polling |
| Token generation | Custom random | `crypto/rand` + `hex.EncodeToString` | Already used in `generateHexToken(16)` |

**Key insight:** Everything needed for this phase already exists in the codebase. GetStatus is implemented, token auth exists, the state machine supports all transitions. The work is purely wiring: connecting existing components and removing the LocalhostOnly barrier.

## Common Pitfalls

### Pitfall 1: Goroutine Leak from Polling Loop
**What goes wrong:** `progressStatus` goroutine polls indefinitely if the instance is terminated while still in `provisioning` state (e.g., user calls DELETE during boot).
**Why it happens:** Polling loop doesn't check if instance has been concurrently terminated.
**How to avoid:** Check the current instance status from DB on each poll iteration. If status is no longer `provisioning` (e.g., someone terminated it), exit the loop immediately. Also enforce a hard timeout (10 minutes).
**Warning signs:** Goroutine count grows over time; leaked goroutines visible in pprof.

### Pitfall 2: Race Between Polling and Ready Callback
**What goes wrong:** `progressStatus` transitions `provisioning -> booting`, but the ready callback arrives almost simultaneously. If both try to update status concurrently, one might fail.
**Why it happens:** Cloud-init execution can be very fast; the callback may arrive within the same second as the booting transition.
**How to avoid:** Both operations use optimistic locking (`UPDATE ... WHERE status = ?`). `progressStatus` sets `provisioning -> booting`, `handleInstanceReady` sets `booting -> running`. They are sequential by state machine design. The only risk is if the callback arrives before the polling detects "running", but that's impossible since the provider must report "running" before cloud-init can execute.

### Pitfall 3: RunPod "RUNNING" Doesn't Mean Cloud-Init Complete
**What goes wrong:** RunPod reports `desiredStatus=RUNNING` when the pod container starts, but cloud-init (the `dockerArgs` startup script) may still be executing.
**Why it happens:** RunPod's "RUNNING" means the container is up, not that the startup script has finished.
**How to avoid:** This is exactly why we have the two-step transition: `provisioning -> booting` (pod running, scripts executing) and `booting -> running` (cloud-init callback received, instance ready). The `booting` state correctly represents "pod is up but not yet ready."

### Pitfall 4: Callback URL Hostname Resolution
**What goes wrong:** The callback URL uses the instance's branded hostname (`gpu-xxxx.gpu.ai`) which may not resolve in DNS if DNS records aren't configured.
**Why it happens:** The callback URL is set to `https://{hostname}/internal/instances/{id}/ready` where hostname is `gpu-xxxx.gpu.ai`.
**How to avoid:** The callback URL should use the gpuctl server's actual public endpoint, not the instance hostname. Currently the code builds: `fmt.Sprintf("https://%s/internal/instances/%s/ready", hostname, instanceID)` where `hostname` is `instanceID + ".gpu.ai"`. This needs to be changed to use the server's actual reachable address. Add a `GPUCTL_PUBLIC_URL` config var or use the WG proxy endpoint. **This is a critical finding -- the callback URL is broken regardless of the LocalhostOnly fix.**

### Pitfall 5: Missing UpdateInstanceIP
**What goes wrong:** When `GetStatus` returns the upstream IP address, we should update the instance's `upstream_ip` in the database so the WireGuard setup can use it later.
**Why it happens:** The current `progressStatus` doesn't store any data from `GetStatus` responses.
**How to avoid:** Update `upstream_ip` in the database when `GetStatus` returns an IP. This is nice-to-have for this phase but important for debugging.

## Open Questions

1. **Callback URL hostname resolution**
   - What we know: Current code builds callback URL as `https://gpu-xxxx.gpu.ai/internal/instances/{id}/ready`. This hostname is the instance's branded hostname, not the gpuctl server's address.
   - What's unclear: Whether DNS is configured to resolve `*.gpu.ai` to the gpuctl server. In development, it certainly isn't.
   - Recommendation: Add a `GPUCTL_PUBLIC_URL` environment variable (e.g., `https://api.gpu.ai`) and use it for callback URLs instead of the instance hostname. This is a prerequisite for the ready callback to work. If omitted, fall back to building from the existing config. This should be addressed in this phase since the callback is broken without it.

2. **Poll interval tuning**
   - What we know: RunPod pods typically boot in 15-60 seconds. API rate limit is generous (~100 req/s).
   - What's unclear: Optimal poll interval for user experience vs API load when many instances are provisioning simultaneously.
   - Recommendation: Start with 5-second interval. This means 3-12 polls per instance boot. Even with 100 concurrent provisions, that's 20 req/s -- well within RunPod's limits.

3. **SSE event for booting transition**
   - What we know: The ready callback publishes an SSE event when transitioning booting->running. No event is published for provisioning->booting.
   - What's unclear: Whether connected SSE clients need to see the booting transition.
   - Recommendation: Publish SSE event for booting transition too. This requires either passing the StatusBroker to the engine or using a callback function. A simple callback function (`OnStatusChange func(instanceID, status string)`) is the cleanest approach.
