<objective>
Implement the two core runtime components of the privacy layer: (1) PostgreSQL-backed IPAM that allocates unique tunnel IPs from the 10.0.0.0/16 subnet with advisory lock protection against races, and (2) the WireGuard Manager that programmatically adds/removes peers on the proxy server via wgctrl-go and manages corresponding iptables port-mapping rules for SSH routing.

Purpose: IPAM provides the tunnel address for each new instance. The Manager wires the WireGuard peer and SSH port-mapping rule so customer connections route through the proxy with a branded hostname. Together they implement the runtime plumbing of the privacy layer.

Output: ipam.go, manager.go, and comprehensive unit tests for both.
</objective>

    **IPAM implementation** (`internal/wireguard/ipam.go`):

    Define an interface for database operations so IPAM is unit-testable without a real database:

    ```go
    // IPAMStore abstracts the database operations needed by IPAM.
    type IPAMStore interface {
        // AcquireAdvisoryLock acquires a transaction-scoped advisory lock.
        AcquireAdvisoryLock(ctx context.Context, tx pgx.Tx, lockID int64) error
        // QueryMaxAddress returns the current maximum wg_address from instances, or nil if none.
        QueryMaxAddress(ctx context.Context, tx pgx.Tx) (net.IP, error)
    }
    ```

    Implement `IPAM` struct:
    ```go
    type IPAM struct {
        subnet    *net.IPNet    // 10.0.0.0/16
        proxyAddr net.IP        // 10.0.0.1 (reserved)
        logger    *slog.Logger
    }
    ```

    Key functions:
    1. `NewIPAM(subnetCIDR string, logger *slog.Logger) (*IPAM, error)` -- Parses the CIDR (default "10.0.0.0/16"), validates it, reserves 10.0.0.1 for proxy.

    2. `AllocateAddress(ctx context.Context, tx pgx.Tx) (net.IP, error)` -- The core allocation function:
       - Execute `SELECT pg_advisory_xact_lock($1)` with constant lock ID `int64(0x475055414950414D)` (represents "GPUAIPAM")
       - Execute `SELECT COALESCE(host(MAX(wg_address))::text, '10.0.0.1') FROM instances WHERE wg_address IS NOT NULL` to get the current max
       - Call `incrementIP()` on the result
       - Validate the next IP is still within the subnet via `ipam.subnet.Contains(nextIP)`
       - Return the next IP (caller inserts it into the instance row within the same transaction, so the advisory lock protects the window)
       - If subnet exhausted, return a descriptive error

       Note: `tx` is a `pgx.Tx` passed in by the caller. The advisory lock is transaction-scoped (`pg_advisory_xact_lock`), so it auto-releases on commit/rollback.

    3. `incrementIP(ip net.IP) net.IP` -- Unexported helper:
       - Copy the IP to avoid mutation
       - Use `ip.To4()` for IPv4
       - Increment from last byte, carrying over
       - Return the incremented IP

    4. `IsProxyAddress(ip net.IP) bool` -- Returns true if the given IP is the reserved proxy address (10.0.0.1).

    5. `SubnetCIDR() string` -- Returns the subnet CIDR string for use in WireGuard AllowedIPs config.

    Use `log/slog` for structured logging. Use `net`, `fmt`, `context` from stdlib. Use `github.com/jackc/pgx/v5` for the Tx type.

    **IPAM tests** (`internal/wireguard/ipam_test.go`):

    Since we cannot use a real database in unit tests, test the IP math and validation logic directly:

    1. `TestNewIPAM` -- Creates IPAM with "10.0.0.0/16", verifies subnet parsed correctly
    2. `TestNewIPAMInvalidCIDR` -- Passes invalid CIDR, expects error
    3. `TestIncrementIP` -- Tests incrementIP with various inputs:
       - 10.0.0.1 -> 10.0.0.2
       - 10.0.0.254 -> 10.0.0.255
       - 10.0.0.255 -> 10.0.1.0 (carry)
       - 10.0.255.255 -> 10.1.0.0 (double carry)
    4. `TestIsProxyAddress` -- 10.0.0.1 returns true, 10.0.0.2 returns false
    5. `TestSubnetCIDR` -- Returns "10.0.0.0/16"
    6. `TestAllocateAddressWithMockTx` -- Create a mock that simulates the DB queries (mock the pgx.Tx interface with a struct that returns predetermined values). Test that:

       - When max address is 10.0.255.254, next is 10.0.255.255 (near exhaustion)
    7. `TestAllocateAddressSubnetExhausted` -- Mock returns 10.0.255.255 as max, verify error about subnet exhaustion

    For mocking pgx.Tx: create a minimal `mockTx` struct that implements just the `Exec` and `QueryRow` methods needed. Use a `maxAddr` field to control what the mock returns. This avoids importing a mocking library.

    **Manager implementation** (`internal/wireguard/manager.go`):

    Replace the existing TODO stub entirely. The Manager is the all-in-one component: it manages WireGuard peers AND iptables port-mapping rules.

    Define a `WGClient` interface for testability (wraps wgctrl.Client):
    ```go
    // WGClient abstracts WireGuard device control for testability.
    type WGClient interface {
        ConfigureDevice(name string, cfg wgtypes.Config) error
        Device(name string) (*wgtypes.Device, error)
        Close() error
    }
    ```

    Define a `CommandRunner` interface for iptables (wraps os/exec):
    ```go
    // CommandRunner abstracts shell command execution for testability.
    type CommandRunner interface {
        Run(ctx context.Context, name string, args ...string) error
    }
    ```

    Implement `execRunner` as the production CommandRunner using `os/exec.CommandContext`.

    Implement `Manager` struct:
    ```go
    type Manager struct {
        client        WGClient
        runner        CommandRunner
        interfaceName string        // e.g., "wg0"
        logger        *slog.Logger
    }
    ```

    Key functions:

    1. `NewManager(client WGClient, interfaceName string, logger *slog.Logger) *Manager` -- Constructor. Uses `execRunner` by default if no custom runner. Actually, accept the runner as parameter for testability: `NewManager(client WGClient, runner CommandRunner, interfaceName string, logger *slog.Logger)`.

    2. `AddPeer(ctx context.Context, publicKeyBase64 string, tunnelIP net.IP, externalPort int) error` -- Per user decision: when adding a peer, also add the iptables rule. Steps:
       a. Parse public key via `wgtypes.ParseKey(publicKeyBase64)`
       b. Configure WireGuard peer via `client.ConfigureDevice(interfaceName, cfg)` with:
          - `PublicKey`: parsed key
          - `ReplaceAllowedIPs`: true
          - `AllowedIPs`: `[]net.IPNet{{IP: tunnelIP, Mask: net.CIDRMask(32, 32)}}`
          - `PersistentKeepaliveInterval`: 25 seconds
       c. Add iptables DNAT rule: `iptables -t nat -A PREROUTING -p tcp --dport {externalPort} -j DNAT --to-destination {tunnelIP}:22`
       d. Add iptables FORWARD rule: `iptables -A FORWARD -p tcp -d {tunnelIP} --dport 22 -j ACCEPT`
       e. If WireGuard peer add succeeds but iptables fails, attempt to remove the peer (rollback). Log the error.

       The `externalPort` is derived by the caller from the tunnel IP. For example, 10.0.0.5 maps to port 10005 (10000 + last octet). The Manager does not derive this -- it receives it as a parameter.

    3. `RemovePeer(ctx context.Context, publicKeyBase64 string, tunnelIP net.IP, externalPort int) error` -- Per user decision: when removing a peer, also remove the iptables rule. Steps:
       a. Parse public key via `wgtypes.ParseKey(publicKeyBase64)`
       b. Remove iptables DNAT rule: `iptables -t nat -D PREROUTING -p tcp --dport {externalPort} -j DNAT --to-destination {tunnelIP}:22`
       c. Remove iptables FORWARD rule: `iptables -D FORWARD -p tcp -d {tunnelIP} --dport 22 -j ACCEPT`
       d. Remove WireGuard peer via `client.ConfigureDevice(interfaceName, cfg)` with `Remove: true`
       e. Log but do not fail if iptables removal fails (peer may have been partially set up)

    4. `ListPeers(ctx context.Context) ([]Peer, error)` -- Calls `client.Device(interfaceName)` and converts the device's peers to our `Peer` type.

    Define `Peer` struct in the same file:
    ```go
    type Peer struct {
        PublicKey     string
        AllowedIPs    []string
        LastHandshake time.Time
        TransferRx    int64
        TransferTx    int64
    }
    ```

    5. `PortFromTunnelIP(ip net.IP) int` -- Exported helper. Derives the external SSH port from the tunnel IP. Formula: `10000 + int(ip.To4()[2])*256 + int(ip.To4()[3])`. This maps the full /16 range (10.0.0.2 to 10.0.255.255) to ports 10002-75535. For simple cases: 10.0.0.5 -> 10005, 10.0.1.0 -> 10256.

    Use structured logging via `log/slog`. Import `golang.zx2c4.com/wireguard/wgctrl/wgtypes`, `net`, `context`, `fmt`, `os/exec`, `time`.
