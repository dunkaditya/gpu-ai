# Phase 4.3: Auth & Idempotency Edge Cases - Research

**Researched:** 2026-02-25
**Domain:** PostgreSQL schema constraints, HTTP middleware ordering, Go concurrency patterns
**Confidence:** HIGH

## Summary

Phase 4.3 addresses two structural bugs discovered during audit: (1) the `email UNIQUE` constraint on the `users` table that prevents multiple Clerk users from being created within the same organization (or across organizations), and (2) a race condition in the `IdempotencyMiddleware` that fails with a 500 error on a new organization's very first POST request because the middleware tries to resolve the org UUID before the handler creates the org record.

Both bugs are well-understood with clear root causes in the existing code. Bug 1 is a schema constraint problem -- the `email` column in `users` (defined in v0 schema) has `UNIQUE NOT NULL`, but `EnsureOrgAndUser` passes an empty string `""` for email, meaning the second user ever created fails the unique constraint. Even if email were populated from Clerk JWTs, two users across different orgs sharing the same email would conflict. Bug 2 is a middleware ordering problem -- the idempotency middleware calls `GetOrgIDByClerkOrgID` which returns `ErrNotFound` for a brand-new org, and the middleware treats this as an internal error (500), never reaching the handler's `EnsureOrgAndUser` that would create the org.

**Primary recommendation:** Fix both bugs with a single migration (drop email UNIQUE constraint, make email nullable) and restructure the idempotency middleware to handle the "org not yet created" case by either deferring org resolution or ensuring the org exists before idempotency key lookup.

## Architecture Patterns

### Pattern 1: Handle ErrNotFound in Idempotency Middleware (Pass-Through)
**What:** When `GetOrgIDByClerkOrgID` returns `ErrNotFound`, the middleware skips idempotency enforcement and passes through to the handler. The handler's `EnsureOrgAndUser` creates the org. Subsequent requests with the same idempotency key will find the org and work normally.
**When to use:** When the idempotency middleware runs before the handler that creates the org.
**Why this is safe:** The idempotency key is scoped to (org_id, key). If the org doesn't exist yet, there cannot be a duplicate key. The first request creates the org, creates the instance, and the response is returned. If the client retries with the same idempotency key, the org now exists and the middleware can enforce deduplication normally. There is a small window for a true race (two concurrent first requests for a new org with the same idempotency key) but this is an extreme edge case and at worst results in a duplicate instance, which is the same behavior as not sending an idempotency key at all.
**Example:**
```go
// In IdempotencyMiddleware, after GetOrgIDByClerkOrgID fails:
internalOrgID, err := dbPool.GetOrgIDByClerkOrgID(ctx, claims.OrgID)
if err != nil {
    if errors.Is(err, db.ErrNotFound) {
        // Org not created yet -- skip idempotency enforcement.
        // The handler will create the org via EnsureOrgAndUser.
        // On retry, the org will exist and idempotency will work.
        next.ServeHTTP(w, r)
        return
    }
    // Genuine error -- return 500.
    slog.Error("idempotency: failed to resolve org ID", ...)
    writeProblem(w, http.StatusInternalServerError, ...)
    return
}
```

### Pattern 2: EnsureOrg in Middleware (Eager Creation)
**What:** Instead of skipping, the idempotency middleware itself calls `EnsureOrgAndUser` (or a simpler `EnsureOrg`) to create the org if it doesn't exist, then proceeds with normal idempotency logic.
**When to use:** When you want idempotency to work even on the very first request for a new org (no pass-through gap).
**Why this is more complex:** Requires either duplicating the `EnsureOrgAndUser` call or extracting an `EnsureOrg` function. Also requires the middleware to know about user claims to call EnsureOrg. Adds complexity for minimal benefit since the concurrent first-request race is extremely unlikely.
**Example:**
```go
internalOrgID, err := dbPool.GetOrgIDByClerkOrgID(ctx, claims.OrgID)
if err != nil {
    if errors.Is(err, db.ErrNotFound) {
        // Create org eagerly so idempotency key can be scoped.
        internalOrgID, err = dbPool.EnsureOrg(ctx, claims.OrgID)
        if err != nil {
            slog.Error("idempotency: failed to ensure org", ...)
            writeProblem(w, http.StatusInternalServerError, ...)
            return
        }
        // Fall through to normal idempotency logic.
    } else {
        slog.Error("idempotency: failed to resolve org ID", ...)
        writeProblem(w, http.StatusInternalServerError, ...)
        return
    }
}
```

### Recommended Approach: Pattern 2 (Eager Creation)

Pattern 2 is recommended because it provides correct idempotency even on the first request for a new org. Extract an `EnsureOrg` function from `EnsureOrgAndUser` that only handles the organization upsert (not user). This keeps the middleware focused and avoids the pass-through gap of Pattern 1.

The `EnsureOrg` function would be:
```go
func (p *Pool) EnsureOrg(ctx context.Context, clerkOrgID string) (string, error) {
    _, err := p.pool.Exec(ctx,
        `INSERT INTO organizations (organization_id, name, clerk_org_id)
         VALUES (gen_random_uuid(), $1, $1)
         ON CONFLICT (clerk_org_id) DO NOTHING`,
        clerkOrgID,
    )
    if err != nil {
        return "", err
    }
    var orgUUID string
    err = p.pool.QueryRow(ctx,
        `SELECT organization_id FROM organizations WHERE clerk_org_id = $1`,
        clerkOrgID,
    ).Scan(&orgUUID)
    return orgUUID, err
}
```

Then `EnsureOrgAndUser` can call `EnsureOrg` internally to avoid duplication.

### Pattern 3: Email Column Fix
**What:** Drop the `UNIQUE` constraint from `users.email` and make the column nullable.
**Why:** The email field serves an informational role in GPU.ai. Clerk manages user identity and email uniqueness in its own system. GPU.ai's users table maps Clerk user IDs to internal org membership. The `clerk_user_id UNIQUE` constraint already prevents duplicate user records. Making email nullable handles the case where `EnsureOrgAndUser` is called with an empty email (current behavior in `handleCreateInstance`).
**Migration:**
```sql
-- Drop the UNIQUE constraint on users.email.
-- The constraint was inherited from v0 schema when email was the login identifier.
-- Now Clerk manages identity; clerk_user_id UNIQUE is the dedup key.
ALTER TABLE users DROP CONSTRAINT users_email_key;

-- Make email nullable since Clerk JWT may not always include email.
ALTER TABLE users ALTER COLUMN email DROP NOT NULL;
```

### Anti-Patterns to Avoid
- **Composite UNIQUE (org_id, email):** Tempting but wrong. Email is not a GPU.ai identity column -- Clerk owns identity. Adding composite uniqueness would still break when two users across different orgs share an email but the constraint isn't the right boundary.
- **Storing email as empty string:** The current code passes `""` to `EnsureOrgAndUser`. Even after dropping UNIQUE, empty strings are misleading. Better to pass `nil`/`NULL` when email is not available and populate from Clerk webhook or JWT claims when possible.
- **Moving idempotency logic into the handler:** Breaks the middleware pattern established by the project. The idempotency middleware is cleanly separated and reusable across routes.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Org resolution race | Custom locking in middleware | `INSERT ... ON CONFLICT DO NOTHING` + `SELECT` pattern (already used in EnsureOrgAndUser) | PostgreSQL handles the race atomically; no need for application-level locks |
| Schema migration | Manual ALTER TABLE in code | SQL migration file via `tools/migrate.py` | Consistent with existing 4-migration approach; migration runner handles transactions |
| Idempotency key cleanup for orphaned keys | Custom reaper goroutine | `CleanupIdempotencyKeys` with `olderThan` duration (already exists in `db/idempotency.go`) | The cleanup function already exists and handles stale keys |

**Key insight:** Both bugs are fixed by straightforward SQL changes and error handling adjustments. No new libraries, no new patterns, no new concurrency primitives needed.

## Common Pitfalls

### Pitfall 1: Forgetting to Drop the Email Index Too
**What goes wrong:** The v0 schema creates `CREATE INDEX idx_users_email ON users(email)` and the v1 migration recreates it. Dropping the UNIQUE constraint (`users_email_key`) removes the unique index but the regular `idx_users_email` index remains. This is fine (it's just a non-unique index for lookups) but if someone later tries to add UNIQUE back, the duplicate index wastes space.
**Why it happens:** PostgreSQL creates an implicit unique index for UNIQUE constraints AND the explicit index was created separately.
**How to avoid:** Drop the explicit `idx_users_email` index in the migration too, since email lookups without uniqueness are unlikely to be a hot query path. Or leave it -- it's harmless but redundant.
**Warning signs:** `\d users` in psql shows two email indexes.

### Pitfall 2: Idempotency Key Without Org -- Incomplete Response Storage
**What goes wrong:** If Pattern 1 (pass-through) is used, the first request creates the instance but the idempotency key is never stored (since the middleware was bypassed). A retry would create a second instance.
**Why it happens:** The middleware skip means no response capture occurs.
**How to avoid:** Use Pattern 2 (eager org creation) so the idempotency key is always stored.
**Warning signs:** Duplicate instances on retry for first-time organizations.

### Pitfall 3: EnsureOrgAndUser Called Twice with Different User IDs
**What goes wrong:** After fixing the email UNIQUE constraint, `EnsureOrgAndUser` can successfully create multiple users in the same org. But the `user_id` passed to `CreateInstance` is the Clerk user ID (a string from JWT), not the internal UUID. Currently the engine uses `req.UserID` which comes from `claims.UserID` (the Clerk subject), and the `users` table's `user_id` is a UUID.
**Why it happens:** The `instances.user_id` FK references `users.user_id` (UUID). The handler passes `claims.UserID` (Clerk user ID, not UUID) as `req.UserID`. But looking at `EnsureOrgAndUser`, it returns the org UUID, not the user UUID. The handler uses the org UUID for `provReq.OrgID` but passes `claims.UserID` for `provReq.UserID` -- this is the Clerk user ID string, not the internal user UUID.
**How to avoid:** `EnsureOrgAndUser` should also return the internal user UUID, or a separate lookup should resolve it. This is technically an existing bug in the `handleCreateInstance` flow but may not have surfaced because only one user per org has been tested so far.
**Warning signs:** Foreign key violation on `instances.user_id` when the Clerk user ID doesn't match a UUID format.

### Pitfall 4: Concurrent First Requests from Same New Org
**What goes wrong:** Two requests arrive simultaneously for a brand-new org. Both call `EnsureOrg` -- the `INSERT ... ON CONFLICT DO NOTHING` handles this correctly (one inserts, one no-ops, both can SELECT the UUID). The idempotency key creation uses a PK on `(org_id, idempotency_key)` which also handles the race. No actual pitfall here -- the existing upsert patterns are race-safe.
**Why it matters:** Confirming this is safe avoids unnecessary complexity.

### EnsureOrgAndUser Refactored to Use EnsureOrg
```go
func (p *Pool) EnsureOrgAndUser(ctx context.Context, clerkOrgID, clerkUserID, email string) (string, error) {
    // Reuse EnsureOrg for the organization upsert.
    orgUUID, err := p.EnsureOrg(ctx, clerkOrgID)
    if err != nil {
        return "", err
    }

    // Upsert user: create if not exists, otherwise update email.
    _, err = p.pool.Exec(ctx,
        `INSERT INTO users (user_id, org_id, clerk_user_id, email, name, role)
         VALUES (gen_random_uuid(), $1, $2, NULLIF($3, ''), '', 'member')
         ON CONFLICT (clerk_user_id) DO UPDATE SET email = NULLIF(EXCLUDED.email, '')`,
        orgUUID, clerkUserID, email,
    )
    if err != nil {
        return "", err
    }

    return orgUUID, nil
}
```

### EnsureOrgAndUser Returns User UUID (Bug Fix for Pitfall 3)
```go
// EnsureOrgAndUser should return both org UUID and user UUID.
func (p *Pool) EnsureOrgAndUser(ctx context.Context, clerkOrgID, clerkUserID, email string) (orgID string, userID string, err error) {
    orgUUID, err := p.EnsureOrg(ctx, clerkOrgID)
    if err != nil {
        return "", "", err
    }

    // Upsert user and return the internal user UUID.
    var userUUID string
    err = p.pool.QueryRow(ctx,
        `INSERT INTO users (user_id, org_id, clerk_user_id, email, name, role)
         VALUES (gen_random_uuid(), $1, $2, NULLIF($3, ''), '', 'member')
         ON CONFLICT (clerk_user_id) DO UPDATE SET email = NULLIF(EXCLUDED.email, '')
         RETURNING user_id`,
        orgUUID, clerkUserID, email,
    ).Scan(&userUUID)
    if err != nil {
        return "", "", err
    }

    return orgUUID, userUUID, nil
}
```

## Open Questions

1. **Should EnsureOrgAndUser return the user UUID?**
   - What we know: `handleCreateInstance` passes `claims.UserID` (Clerk subject) as `req.UserID`, which gets stored in `instances.user_id`. But `instances.user_id` is an FK to `users.user_id` (a UUID). If the Clerk subject happens to be a UUID-compatible string, it would fail the FK check anyway since it doesn't match the auto-generated user UUID.
   - What's unclear: Whether this bug has surfaced in testing (it would only fail if FK constraints are enforced and the Clerk user ID doesn't match a user UUID in the DB).
   - Recommendation: Fix in this phase by having `EnsureOrgAndUser` return the user UUID and updating `handleCreateInstance` to use it. This is directly related to AUTH-03 (multi-user org support) and is a natural scope extension. **Confidence: HIGH** that this is a real bug based on code tracing.

2. **Should the email column be populated from Clerk JWT claims?**
   - What we know: The Clerk JWT `SessionClaims` struct includes email-related fields. Currently the handler passes `""` for email.
   - What's unclear: Whether Clerk session claims reliably include email for all auth configurations.
   - Recommendation: Make email nullable (NULL instead of empty string). Defer email population to a Clerk webhook integration or dashboard profile sync (not in scope for this phase). **Confidence: MEDIUM** -- Clerk JWT structure may vary by configuration.
