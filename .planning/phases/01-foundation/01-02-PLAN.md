<task type="auto">
  <name>Task 2: Implement Python migration runner</name>
  <files>tools/migrate.py</files>
  <action>
Replace the TODO stub in `tools/migrate.py` with a working implementation using psycopg2 and click.

1. **CLI interface** (using click):
   - `python tools/migrate.py` — apply all pending migrations
   - `python tools/migrate.py --status` — show table of applied vs pending migrations
   - `python tools/migrate.py --target VERSION` — migrate up to a specific version filename

2. **Core logic**:
   - Read `DATABASE_URL` from environment variable. If not set, print error and exit 1.
   - Connect to PostgreSQL using `psycopg2.connect(os.environ["DATABASE_URL"])`
   - Create tracking table if not exists:
     ```sql
     CREATE TABLE IF NOT EXISTS schema_migrations (
         version VARCHAR(255) PRIMARY KEY,
         applied_at TIMESTAMPTZ DEFAULT NOW()
     )
     ```
   - Scan `database/migrations/*.sql` files, sorted by filename (date-prefix ensures correct order)
   - Query `schema_migrations` for already-applied versions
   - For each pending migration:
     - Read the .sql file content
     - Execute the SQL within a transaction (each migration is its own transaction)
     - Insert the version (filename) into `schema_migrations`
     - Print: `Applied: {filename}`
   - If no pending migrations, print: `No pending migrations`

3. **--status flag**:
   - Use `tabulate` to display a table with columns: Version, Status, Applied At
   - Applied migrations show date, pending show "pending"

4. **--target flag**:
   - Only apply migrations up to and including the target version filename
   - Error if target version not found in migration files

5. **Error handling**:
   - If a migration fails, the transaction rolls back and the script exits with error
   - Print the SQL error and which migration file failed

6. Use `#!/usr/bin/env python3` shebang. Add docstring matching the existing one.

Reference: `tools/requirements.txt` already has psycopg2-binary, click, tabulate.
