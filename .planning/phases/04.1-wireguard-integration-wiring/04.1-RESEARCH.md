# Phase 4.1: WireGuard Integration Wiring - Research

**Researched:** 2026-02-24
**Domain:** WireGuard integration wiring (Go, wgctrl-go, RunPod GraphQL API)
**Confidence:** HIGH

## Summary

Phase 4.1 is a gap closure phase that wires together already-implemented but disconnected WireGuard components. The audit (`v1.0-MILESTONE-AUDIT.md`) identified that all privacy layer code exists and passes tests in isolation, but the components are never initialized or connected in the production binary. Specifically: (1) `wireguard.NewManager` and `wireguard.NewIPAM` are never called in `main.go`, so `EngineDeps.WGManager` and `EngineDeps.IPAM` are always nil, (2) `wireguard.Manager.AddPeer` is never called during provisioning even though the engine generates keys and renders cloud-init, and (3) the RunPod adapter silently discards the `ProvisionRequest.StartupScript` field.

All building blocks exist. The `wireguard` package has `NewManager`, `NewIPAM`, `GenerateKeyPair`, `EncryptPrivateKey`, `RenderBootstrap`, `AddPeer`, `RemovePeer`, and `PortFromTunnelIP` -- all tested. The `provision.Engine` already has WG-conditional code at `engine.go:149-202` that generates keys, allocates IPAM, and renders cloud-init. The `provider.ProvisionRequest` already has a `StartupScript` field. The `config.Config` already has `WGInterfaceName`, `WGProxyEndpoint`, `WGProxyPublicKey`, and `WGEncryptionKeyBytes`. This phase requires only wiring code -- no new algorithms, no new packages, no new APIs.

**Primary recommendation:** Wire three specific connection points: (1) initialize WG Manager + IPAM in `main.go` and pass to `EngineDeps`, (2) add `AddPeer` call in `engine.Provision` after IPAM allocation, and (3) pass `req.StartupScript` as `dockerArgs` in the RunPod adapter's GraphQL mutations. Make WG initialization conditional on config presence so the binary still works without WG config for development.

## Architecture Patterns

### Recommended Changes (3 files)

```
cmd/gpuctl/main.go              # ADD: WG Manager + IPAM initialization (6 lines)
internal/provision/engine.go     # ADD: AddPeer call after IPAM allocation (10 lines)
```

### Pattern 1: Conditional WG Initialization in main.go
**What:** Initialize WireGuard Manager and IPAM only when WG config env vars are present. Pass nil when unconfigured (existing pattern from Phase 4's engine design).
**When to use:** Always -- the binary must work without WG for development.
**Example:**
```go
// In main.go, after provider registry setup:
var wgMgr *wireguard.Manager
var ipam *wireguard.IPAM

if cfg.WGEncryptionKeyBytes != nil {
    // Create wgctrl client for the WireGuard interface.
    wgClient, err := wgctrl.New()
    if err != nil {
        slog.Error("failed to create WireGuard client", "error", err)
        os.Exit(1)
    }
    defer wgClient.Close()

    wgMgr = wireguard.NewManager(wgClient, nil, cfg.WGInterfaceName, logger)
    slog.Info("wireguard manager initialized", "interface", cfg.WGInterfaceName)

    ipam, err = wireguard.NewIPAM("10.0.0.0/16", logger)
    if err != nil {
        slog.Error("failed to initialize IPAM", "error", err)
        os.Exit(1)
    }
    slog.Info("wireguard IPAM initialized", "subnet", "10.0.0.0/16")
}

// Then pass to EngineDeps:
engine := provision.NewEngine(provision.EngineDeps{
    Registry:  providerRegistry,
    DB:        dbPool,
    Config:    cfg,
    Logger:    logger,
    WGManager: wgMgr,
    IPAM:      ipam,
})
```

### Pattern 2: AddPeer in Provision Flow
**What:** After IPAM allocates a tunnel IP, call `AddPeer` on the WG Manager to register the peer on the proxy server before calling the provider.
**When to use:** During provisioning, after key generation and IPAM allocation, before the provider.Provision call.
**Example:**
```go
// In engine.go Provision(), after IPAM allocation (line ~179) and before provider call:
if e.wgMgr != nil && wgAddress != nil {
    addrStr, _, _ := strings.Cut(*wgAddress, "/")
    tunnelIP := net.ParseIP(addrStr)
    if tunnelIP == nil {
        return nil, fmt.Errorf("provision: parse tunnel IP %q: invalid", *wgAddress)
    }
    externalPort := wireguard.PortFromTunnelIP(tunnelIP)
    if err := e.wgMgr.AddPeer(ctx, kp.PublicKey, tunnelIP, externalPort); err != nil {
        return nil, fmt.Errorf("provision: add WireGuard peer: %w", err)
    }
    e.logger.Info("added WireGuard peer for instance",
        slog.String("instance_id", instanceID),
        slog.String("tunnel_ip", tunnelIP.String()),
        slog.Int("external_port", externalPort),
    )
}
```

### Pattern 3: RunPod StartupScript as dockerArgs
**What:** Pass `ProvisionRequest.StartupScript` as the `dockerArgs` field in RunPod GraphQL mutations. RunPod's `dockerArgs` overrides the container's CMD/entrypoint.
**When to use:** In both `provisionOnDemand` and `provisionSpot` methods.
**Example:**
```go
// In adapter.go provisionOnDemand() and provisionSpot():
input := map[string]any{
    "cloudType":         "SECURE",
    "gpuCount":          gpuCount,
    "gpuTypeId":         gpuTypeID,
    "name":              name,
    "imageName":         imageName,
    "containerDiskInGb": defaultContainerDisk,
    "volumeInGb":        0,
    "startSsh":          true,
    "ports":             "22/tcp",
    "env":               env,
    "dockerArgs":        startupScript,  // <-- NEW: delivers bootstrap script
}
```

### Anti-Patterns to Avoid
- **Initializing WG Manager when config is missing:** The binary must remain runnable without WG config for local development. Keep the nil-guard pattern.
- **Adding AddPeer after the provider.Provision call:** The peer must be registered on the proxy BEFORE the instance boots, so when the instance runs the bootstrap script and tries to establish the WG tunnel, the proxy already knows about it.
- **Passing StartupScript as an env var instead of dockerArgs:** Env vars have size limits and the bootstrap script is substantial (167 lines). `dockerArgs` is RunPod's documented mechanism for custom startup commands.

## Common Pitfalls

### Pitfall 1: wgctrl.New() Requires Root or CAP_NET_ADMIN
**What goes wrong:** `wgctrl.New()` fails with "operation not permitted" when gpuctl runs without root privileges.
**Why it happens:** WireGuard device control requires elevated capabilities on Linux.
**How to avoid:** In production, gpuctl runs on the proxy server with root or `CAP_NET_ADMIN`. For development, the nil-guard pattern means WG is simply skipped. Add a clear log message when WG init fails due to permissions so developers understand it is expected.
**Warning signs:** "failed to create WireGuard client" error on startup in dev.

### Pitfall 2: AddPeer Before Provider vs After Provider
**What goes wrong:** If AddPeer is called after provider.Provision, there is a race window where the instance boots and tries to establish a WG tunnel before the proxy knows about it. The tunnel handshake fails.
**Why it happens:** Cloud-init/bootstrap script runs immediately on pod boot. If the peer is not registered on the proxy, the handshake has no matching peer config.
**How to avoid:** Call AddPeer BEFORE calling provider.Provision. The peer exists on the proxy before the instance even starts booting.
**Warning signs:** WireGuard tunnel established intermittently; some instances connect, others time out on handshake.

### Pitfall 3: AddPeer Failure Leaves Orphaned IPAM Allocation
**What goes wrong:** IPAM allocates an address (committed to DB), then AddPeer fails (WG or iptables error), but the IPAM address is already committed.
**Why it happens:** IPAM allocation commits in its own transaction before AddPeer runs.
**How to avoid:** This is acceptable for now -- the address space is /16 (65,534 addresses). Orphaned addresses from failed provisions are a minor leak. For production, consider moving IPAM allocation into the same transaction as instance creation, or implement address reclamation for failed provisions.
**Warning signs:** Address space exhaustion much faster than expected instance count.

### Pitfall 4: Config Requires WG Env Vars Even When Not Using WG
**What goes wrong:** Current `config.Load()` treats `WG_ENCRYPTION_KEY`, `WG_PROXY_ENDPOINT`, and `WG_PROXY_PUBLIC_KEY` as required. Developers without WG proxy cannot start the binary.
**Why it happens:** Config validation added in Phase 3 made these required without considering dev workflow.
**How to avoid:** Make WG config vars optional. If any are missing, log a warning and skip WG initialization (return nil for WGManager and IPAM). The engine's nil-guard already handles this gracefully.
**Warning signs:** "missing required environment variables: WG_ENCRYPTION_KEY, WG_PROXY_ENDPOINT, WG_PROXY_PUBLIC_KEY" error on startup.

### Pitfall 5: dockerArgs String Quoting in RunPod GraphQL
**What goes wrong:** The bootstrap script contains single quotes, double quotes, and special characters. If improperly escaped in the GraphQL JSON payload, the mutation fails or the script executes incorrectly.
**Why it happens:** The script is a complex bash script with heredocs, variable expansion, and special characters.
**How to avoid:** The script is passed as a Go string value in the `input` map, which gets JSON-marshaled by `json.Marshal()`. JSON encoding handles all escaping automatically. The script content is already properly generated by `RenderBootstrap` which uses `text/template`. No manual escaping needed.
**Warning signs:** RunPod API returning GraphQL errors about invalid input; pod starts but script does not execute.

### Pitfall 6: Bootstrap Script Assumes VM Environment But Runs in Docker Container
**What goes wrong:** The bootstrap script at `templates/bootstrap.sh.tmpl` uses `systemctl`, `hostnamectl`, `apt-get install`, and iptables commands that may fail or behave differently in a Docker container.
**Why it happens:** The script was designed for a VM-based cloud-init model, but RunPod uses Docker containers.
**How to avoid:** This is a known limitation documented in `PITFALLS.md` as Pitfall 1 (HIGH severity). For Phase 4.1, the script is delivered as-is via `dockerArgs`. Many commands (`apt-get install wireguard-tools`, `mkdir`, `cat >`, `chmod`, `curl`) work fine in Docker. The `systemctl` calls have `2>/dev/null || true` fallbacks. `hostnamectl` has a similar fallback. The script is designed with degraded-mode awareness. Full Docker-native redesign is deferred.
**Warning signs:** Pod boots but WireGuard tunnel fails; SSH not reachable through proxy; ready callback not firing.

## Open Questions

1. **WG config optional vs required in config.Load()**
   - What we know: Currently `WG_ENCRYPTION_KEY`, `WG_PROXY_ENDPOINT`, `WG_PROXY_PUBLIC_KEY` are required in `config.Load()`. Making them optional would allow dev without a WG proxy.
   - What's unclear: Whether to change config validation in this phase or a separate fix.
   - Recommendation: Make WG config vars optional in this phase. The engine already handles nil WGManager gracefully. Config should match. This is a small change (remove from `missing` slice, use empty-string defaults) that directly enables the conditional initialization pattern.

2. **wgctrl.New() import and build constraint**
   - What we know: `wgctrl.New()` creates a real system client that needs Linux netlink. On macOS (dev), it may fail to compile or panic.
   - What's unclear: Whether `wgctrl.New()` compiles on macOS without errors.
   - Recommendation: The wgctrl library does compile on macOS (uses a userspace implementation). It will return an error at runtime if no WG interface exists, which the nil-guard handles. The conditional init pattern (`if cfg.WGEncryptionKeyBytes != nil`) means `wgctrl.New()` is only called when WG is configured, which would only be on the production proxy server.

3. **Bootstrap script viability in Docker containers**
   - What we know: The bootstrap script uses `apt-get`, `systemctl`, `iptables` which may not work in all Docker environments. RunPod containers run as root and appear to grant NET_ADMIN.
   - What's unclear: Whether the full script executes successfully on RunPod. This has never been tested.
   - Recommendation: Deliver the script as-is for Phase 4.1. All commands have error-tolerant fallbacks (`|| true`, `2>/dev/null`). A Docker-native redesign is a separate effort tracked in pitfalls. Phase 4.1 focuses on wiring, not script redesign.

4. **AddPeer rollback on provider.Provision failure**
   - What we know: If AddPeer succeeds but provider.Provision fails, there's an orphaned WG peer on the proxy.
   - What's unclear: Whether to add cleanup logic in the Provision error path.
   - Recommendation: Add a best-effort RemovePeer call in the error path after provider.Provision fails. This matches the existing best-effort cleanup pattern used in Terminate. Log but don't fail on cleanup errors.
