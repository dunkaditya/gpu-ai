# Milestone v1.0 Audit Report

## Executive Summary

**Status: GAPS FOUND** -- 6 unsatisfied requirements, 5 partial requirements, 2 broken E2E flows.

All 4 phases (Foundation, Provider Abstraction, Privacy Layer, Auth + Instance Lifecycle) have been implemented with 14 plans executed across 31 minutes total. Code is committed and all plans have SUMMARY.md files confirming implementation.

However, critical integration gaps exist primarily in the **WireGuard privacy layer wiring** and the **instance state machine lifecycle**. The privacy layer components (keygen, IPAM, peer manager, cloud-init template) are implemented and tested in isolation, but never initialized or connected in the production binary. Additionally, the instance state machine has a missing transition that prevents instances from ever reaching "running" status.

**No VERIFICATION.md files exist for any phase** -- formal verification was never performed.

## Critical Integration Gaps

### 1. WireGuard Components Never Initialized in main.go

**Root cause:** `cmd/gpuctl/main.go` creates `EngineDeps` with only `Registry`, `DB`, `Config`, `Logger`. The `WGManager` and `IPAM` fields are omitted (nil).

**Impact:** The guard at `engine.go:149` (`if e.wgMgr != nil`) is always false. All WireGuard operations are skipped: key generation, IPAM allocation, cloud-init rendering, peer addition, and peer cleanup.

**Affected:** PRIV-01, PRIV-02, PRIV-03, PRIV-04, PRIV-05, PRIV-06

### 2. AddPeer Never Called in Provision Path

**Root cause:** `engine.Provision()` generates WG keys and renders cloud-init but never calls `wireguard.Manager.AddPeer` to register the peer on the proxy server. Only `RemovePeer` is called during termination.

**Affected:** PRIV-02

### 3. RunPod Adapter Ignores StartupScript

**Root cause:** `runpod.Adapter.Provision()` reads `req.InstanceID`, `req.CallbackURL`, `req.InternalToken`, and `req.SSHPublicKeys` but never reads `req.StartupScript`. The rendered cloud-init bootstrap script is silently discarded.

**Affected:** PRIV-05, PRIV-06, PRIV-07

### 4. Missing provisioning -> booting State Transition

**Root cause:** `progressStatus` goroutine only transitions `creating -> provisioning`. No code transitions `provisioning -> booting`. `SetInstanceRunning` requires `status='booting'` but instance never reaches that state.

**Affected:** INST-05, INST-07

### 5. Cloud-Init Callback URL Unreachable

**Root cause:** `/internal/instances/{id}/ready` is protected by `LocalhostOnly` middleware. Cloud-init on the GPU instance calls this URL from a non-loopback IP, receives 404.

**Affected:** INST-07

## Broken E2E Flows

### Flow 1: Instance Becoming Running

```
POST /api/v1/instances -> engine.Provision -> progressStatus(creating->provisioning)
    -> [BREAK: no code transitions provisioning->booting]
    -> cloud-init ready callback -> [BREAK: LocalhostOnly blocks network request]
    -> SetInstanceRunning(booting->running) -> [BREAK: never reached]
```

Instance permanently stuck at `"provisioning"` (displayed as `"starting"` to customer).

### Flow 2: WireGuard Privacy Layer

```
Provision request -> [BREAK: wgMgr nil, entire WG block skipped]
    -> generate WG keys -> allocate IPAM -> render cloud-init
    -> [BREAK: RunPod adapter ignores StartupScript]
    -> [BREAK: AddPeer never called on proxy]
    -> instance runs bootstrap -> WG tunnel establishes -> SSH routed through proxy
```

Instances provisioned as bare RunPod containers with no WireGuard tunnel, no branded hostname, no provider scrubbing.

## Structural Issues

### Idempotency Middleware Race with Org Creation

`IdempotencyMiddleware` calls `GetOrgIDByClerkOrgID` before the handler runs. For a new org's first `POST /api/v1/instances` with `Idempotency-Key`, the org doesn't exist yet -> HTTP 500. Without the header, the request works fine since `EnsureOrgAndUser` creates the org in the handler.

**Affected:** API-11

### Email UNIQUE Constraint Blocks Multi-User Orgs

`EnsureOrgAndUser` inserts `email=""`. The `users` table has `UNIQUE NOT NULL` on email. A second distinct Clerk user in the same org triggers a constraint violation on the empty string.

**Affected:** AUTH-03
